```markdown
# CLAUDE - 代码逆向工程与API文档生成规范

This file provides guidance to LLM when performing code reverse engineering and API documentation generation.

### 第一部分：核心工作原则 (Guiding Principles)
这是执行代码逆向分析与API文档生成的顶层思想，指导所有具体行为。

#### 基础分析原则
- **完整性优先 (Completeness First)：** 必须分析当前目录下的全量代码文件，不遗漏任何文件、任何网络请求、任何加密逻辑、任何安全机制。遗漏即失败。
- **精确性至上 (Precision Above All)：** 所有文档化的URL、参数名、请求头字段、Cookie字段、响应字段必须与源代码完全一致，严禁猜测、臆造或模糊化处理。
- **可直接调用 (Directly Callable)：** 最终产出的文档必须达到"任何开发者拿到文档即可直接调用API"的标准，所有代码示例必须可直接运行，无需额外推断。

#### 深度逆向原则
- **追根溯源 (Trace to Origin)：** 对每一个网络请求，必须追踪其参数的来源（硬编码、用户输入、前序请求响应、算法生成、环境变量等），形成完整的数据流链路。
- **算法透明化 (Algorithm Transparency)：** 对所有加密、签名、编码、哈希算法必须还原其完整逻辑，包括密钥来源、盐值、迭代次数、编码方式等全部细节。
- **时序还原 (Sequence Restoration)：** 必须还原API调用的时序关系和依赖链，明确哪些请求必须先于哪些请求执行，哪些参数来源于前序请求的响应。

#### 文档工程原则
- **结构化输出 (Structured Output)：** 最终文档必须按照严格规范的Markdown格式输出，层次分明，索引清晰。
- **三语言覆盖 (Tri-Language Coverage)：** 每个API必须同时提供Python、JavaScript和cURL三种语言的完整可运行代码示例。
- **防歧义设计 (Ambiguity Prevention)：** 对于所有可能产生歧义的字段、参数、行为，必须附加明确的说明和约束条件。

### 第二部分：具体执行指令 (Actionable Instructions)
这是执行代码逆向分析时必须严格遵守的具体操作指南。

#### 沟通与语言规范
- **默认语言：** 请默认使用简体中文进行所有交流、分析过程陈述和文档撰写。
- **代码与术语：** 所有代码实体（URL路径、参数名、请求头字段名、函数名、变量名等）必须保持与源代码一致的原文（通常为英文）。
- **代码示例语言：** 代码示例中的注释使用中文，代码本身使用对应语言的标准写法。
- **文档标题与描述：** 使用中文撰写，确保非英语母语开发者可无障碍阅读。

#### 批判性分析与严谨验证
- **交叉验证：** 对于每一个发现的API端点，必须在源代码中找到明确的调用证据（函数调用、请求构造代码），不允许仅凭URL字符串推测。
- **参数完整性验证：** 对每个请求，必须验证所有参数是否已被完整记录，包括隐藏参数、动态生成参数和条件性参数。
- **响应格式验证：** 响应格式必须基于源代码中的解析逻辑推断，而非凭空编造。如果源代码中无法确定响应格式，必须明确标注"响应格式需通过实际调用确认"。

#### 分析执行策略 (Analysis Execution Strategy)

##### 第一阶段：全量文件扫描
- **文件遍历：** 扫描当前目录及所有子目录下的全部代码文件，包括但不限于：
  - `.py`、`.js`、`.ts`、`.jsx`、`.tsx`、`.java`、`.c`、`.cpp`、`.go`、`.rs`、`.rb`、`.php`
  - `.json`、`.yaml`、`.yml`、`.toml`、`.ini`、`.env`、`.cfg` 等配置文件
  - `.html`、`.xml`、`.css`、`.scss` 等前端文件
  - `Dockerfile`、`docker-compose.yml`、`Makefile`、`package.json`、`requirements.txt`、`go.mod` 等项目管理文件
- **文件分类：** 将扫描到的文件按功能分类：入口文件、路由/控制器文件、模型文件、工具/辅助文件、配置文件、测试文件。
- **依赖识别：** 识别项目依赖的第三方库和框架，记录其版本号和用途。

##### 第二阶段：网络请求深度分析
对每一个发现的网络请求，必须提取并记录以下全部信息：

**请求基本信息：**
| 分析项 | 要求 |
|--------|------|
| 请求URL | 完整URL，包含协议、域名/基地址、路径、查询参数模板 |
| 请求方法 | GET / POST / PUT / DELETE / PATCH / HEAD / OPTIONS |
| Content-Type | 请求体的内容类型（application/json、application/x-www-form-urlencoded、multipart/form-data 等） |
| 请求编码 | 字符编码（UTF-8 等） |

**请求头分析：**
| 分析项 | 要求 |
|--------|------|
| 标准请求头 | User-Agent、Referer、Origin、Accept、Accept-Language、Accept-Encoding 等 |
| 自定义请求头 | 所有非标准的自定义请求头字段（如 X-Token、X-Signature、Authorization 等） |
| 动态请求头 | 标注哪些请求头的值是动态生成的，并说明生成逻辑 |
| 条件请求头 | 标注哪些请求头仅在特定条件下才会携带 |

**请求参数分析：**
| 分析项 | 要求 |
|--------|------|
| 路径参数 | URL路径中的动态参数（如 `/api/users/{id}`） |
| 查询参数 | URL中的查询字符串参数（如 `?page=1&size=20`） |
| 请求体参数 | POST/PUT等请求的Body参数 |
| 参数类型 | 每个参数的数据类型（string、number、boolean、array、object等） |
| 参数必选性 | 每个参数是必填还是选填 |
| 参数默认值 | 如果有默认值，必须标注 |
| 参数取值范围 | 如果有枚举值或取值范围限制，必须列出 |
| 参数来源 | 每个参数值的来源（用户输入、前序API响应、算法计算、配置文件、时间戳等） |

**Cookie分析：**
| 分析项 | 要求 |
|--------|------|
| 必需Cookie | 请求必须携带的Cookie字段 |
| Cookie来源 | 每个Cookie的获取方式（登录响应Set-Cookie、前序请求、手动设置等） |
| Cookie生命周期 | 过期时间、是否持久化 |
| Cookie属性 | Domain、Path、Secure、HttpOnly、SameSite 等属性 |

**响应分析：**
| 分析项 | 要求 |
|--------|------|
| 状态码 | 可能返回的HTTP状态码及其含义 |
| 响应头 | 重要的响应头字段（Set-Cookie、Location等） |
| 响应体格式 | JSON结构、字段名、字段类型、字段含义 |
| 错误响应 | 错误情况下的响应格式和错误码含义 |
| 响应示例 | 成功和失败的响应示例 |

##### 第三阶段：算法与加密机制分析
- **加密算法识别：** 识别所有使用的加密算法（AES、RSA、DES、SM2、SM4 等），记录加密模式（ECB、CBC、GCM等）、填充方式（PKCS5、PKCS7等）、密钥长度。
- **哈希算法识别：** 识别所有哈希算法（MD5、SHA-1、SHA-256、HMAC 等），记录是否加盐、盐值来源、迭代次数。
- **签名算法分析：** 识别所有签名机制，完整记录签名字符串的拼接规则、排序规则、编码方式。
- **编码算法识别：** 识别所有编码方式（Base64、URL编码、Hex编码、Unicode编码等）。
- **密钥管理分析：** 记录所有密钥的来源（硬编码、环境变量、远程获取、动态生成），密钥的格式和长度。
- **混淆/反混淆：** 如果代码经过混淆（JavaScript混淆、字节码等），需要描述混淆方式，并尽力还原核心逻辑。
- **算法完整复现：** 对每个算法提供完整的Python和JavaScript复现代码，确保输入相同数据可得到相同结果。

##### 第四阶段：会话与安全机制分析
- **认证机制：** 完整描述认证流程（账号密码登录、OAuth、JWT、Token刷新等），包括认证凭证的获取、存储、传递和过期处理。
- **会话管理：** 分析Session/Token的生命周期管理，包括创建、续期、销毁机制。
- **防护机制识别：** 
  - CSRF防护（Token验证、Referer检查等）
  - 速率限制/频率控制
  - 验证码机制（图形验证码、滑动验证、短信验证等）
  - 设备指纹/浏览器指纹
  - IP黑白名单
  - 请求签名/时间戳验证
  - WAF规则
- **反爬虫机制：** 如果存在反爬虫策略（User-Agent检测、行为分析、JavaScript渲染要求等），必须详细记录。

##### 第五阶段：调用链路与时序还原
- **API依赖关系图：** 绘制API之间的依赖关系，标明哪些API需要先调用。
- **完整调用流程：** 还原从首次访问到完成核心业务操作的完整API调用时序。
- **状态流转：** 记录会话/Token/Cookie在整个调用链中的流转过程。

##### Python 代码示例规范
```python
# 必须使用 requests 库（如果涉及异步则使用 aiohttp）
# 必须包含完整的请求头
# 必须包含错误处理
# 必须包含响应解析
# 必须包含中文注释说明每一步的作用

import requests

def api_example():
    """API调用示例 - [API名称和用途说明]"""
    
    url = "https://example.com/api/endpoint"
    
    # 请求头（每个字段都要注释说明用途和来源）
    headers = {
        "User-Agent": "Mozilla/5.0 ...",  # 浏览器标识
        "Content-Type": "application/json",  # 请求体格式
        "Authorization": "Bearer {token}",  # 认证令牌，来源于登录接口响应
        "X-Custom-Header": "value",  # 自定义头，用途说明
    }
    
    # Cookie（说明每个Cookie的来源和作用）
    cookies = {
        "session_id": "xxx",  # 会话ID，来源于登录响应的Set-Cookie
    }
    
    # 请求参数（说明每个参数的含义、类型、是否必填）
    payload = {
        "param1": "value1",  # 参数说明 | 类型：string | 必填
        "param2": 123,  # 参数说明 | 类型：number | 选填，默认值：0
    }
    
    try:
        response = requests.post(
            url,
            headers=headers,
            cookies=cookies,
            json=payload,
            timeout=30
        )
        response.raise_for_status()
        
        # 解析响应
        data = response.json()
        print(f"请求成功: {data}")
        return data
        
    except requests.exceptions.RequestException as e:
        print(f"请求失败: {e}")
        raise
```

// 必须同时提供 fetch API 和 axios 版本（优先 fetch）

// 必须包含错误处理
// 必须包含响应解析

// === 使用 fetch API ===
async function apiExample() {
    /** API调用示例 - [API名称和用途说明] */
    
    const url = "https://example.com/api/endpoint";
    
    // 请求头（每个字段都要注释说明用途和来源）
    const headers = {
        "User-Agent": "Mozilla/5.0 ...",  // 浏览器标识
        "Content-Type": "application/json",  // 请求体格式
        "Authorization": "Bearer {token}",  // 认证令牌
    };
    
    // 请求参数
    const payload = {
        param1: "value1",  // 参数说明 | 类型：string | 必填
        param2: 123,  // 参数说明 | 类型：number | 选填
    };
    
    try {
        const response = await fetch(url, {
            method: "POST",
            headers: headers,
            body: JSON.stringify(payload),
            credentials: "include",  // 携带Cookie
        });
        
        if (!response.ok) {
            throw new Error(`HTTP ${response.status}: ${response.statusText}`);
        }
        
        const data = await response.json();
        console.log("请求成功:", data);
        return data;
        
    } catch (error) {
        console.error("请求失败:", error);
        throw error;
    }
}
```

### 第五部分：异常处理与兜底策略 (Exception Handling)
